## 说明

本技能对用户提供的多个候选方案（通常 2～6 个），按照定制化的评估维度建立加权评分矩阵，为每个维度量化打分并分析各方案的核心优势与致命弱点，最终输出可视化的对比报告，帮助团队在讨论中形成共识，避免凭主观偏好做技术决策。

**适用场景**：

- 场景一：技术选型时（如选择消息队列：Kafka vs RabbitMQ vs RocketMQ），需要从性能、可运维性、成本、社区等多维度系统比较，而非仅凭熟悉程度做选择
- 场景二：功能实现有多种技术路径时（如缓存策略：本地缓存 vs Redis vs CDN），需要量化各路径的收益与代价，为团队讨论提供客观依据
- 场景三：产品功能方案评审时，需要在多个用户体验方案之间做取舍（如分页加载 vs 无限滚动 vs 虚拟滚动），需要对比用户体验、实现复杂度和性能影响
- 场景四：架构重构时，对比维持现状、渐进式重构和推倒重写三种策略的成本与风险

**前提条件**：

- 用户需提供至少 2 个明确的候选方案
- 用户应说明决策的核心目标（如"优先考虑运维成本"或"性能是最高优先级"），以便合理分配评估维度权重
- 若有硬性约束（如预算上限、团队技术栈限制），应提前说明，用于过滤不满足条件的方案

**不适用场景**：

- 不适用于只有一个候选方案的场景（无法对比）
- 对比结果基于提供的信息，若某方案的关键数据缺失，对应维度将标注"数据不足"，不影响其他维度的比较

## 流程

1. **整理候选方案**：列出所有候选方案，为每个方案补充必要的背景描述（若用户未提供，根据领域知识补充），确保每个方案的描述达到同等详细程度。

2. **定义评估维度**：根据用户的决策目标和场景，从以下维度库中选取 4～8 个最相关的维度（也可根据场景新增）：

   | 维度类别 | 可选维度 |
   |---------|---------|
   | **技术质量** | 性能（吞吐量/延迟）、可靠性/容错性、安全性、可扩展性 |
   | **工程成本** | 实现复杂度、学习曲线/上手难度、与现有技术栈的兼容性 |
   | **运营成本** | 运维难度、监控可观测性、许可证费用/开源成本 |
   | **生态支持** | 社区活跃度、文档质量、第三方集成数量 |
   | **业务影响** | 上线周期、对用户体验的影响、迁移/回滚风险 |

3. **设置权重**：根据用户提供的优先级，为每个维度分配权重（所有维度权重之和为 100%）。若用户未指定优先级，提出权重建议并请用户确认。

4. **逐方案逐维度打分**：对每个方案的每个维度，给出 1～5 分的评分：

   | 分值 | 含义 |
   |------|------|
   | 5 | 该维度表现极佳，无明显短板 |
   | 4 | 表现良好，有轻微不足 |
   | 3 | 表现中等，满足基本要求 |
   | 2 | 存在明显不足，需要额外投入弥补 |
   | 1 | 表现极差或完全不满足要求 |

   打分时附上简短说明（1～2 句），说明打分依据。

5. **计算加权总分**：对每个方案，按以下公式计算：

   ```
   加权总分 = Σ（维度得分 × 维度权重）
   ```

6. **识别优劣势**：对每个方案，提炼：
   - **核心优势**（得分最高的 2～3 个维度）：该方案真正"赢"在哪里
   - **致命弱点**（得分最低的 1～2 个维度）：什么情况下绝对不应选择此方案
   - **适用场景**：结合优劣势，描述此方案最适合的场景

7. **输出对比报告**：按以下格式输出：

   ```markdown
   ## 方案对比报告

   ### 候选方案概述
   - **方案 A**：……（1～2句描述）
   - **方案 B**：……
   - **方案 C**：……

   ### 评估矩阵

   | 评估维度 | 权重 | 方案 A | 方案 B | 方案 C |
   |---------|------|--------|--------|--------|
   | 性能     | 30%  | 5（说明） | 3（说明） | 4（说明） |
   | 运维难度  | 25%  | 2（说明） | 4（说明） | 3（说明） |
   | 实现复杂度 | 20% | 3（说明） | 4（说明） | 2（说明） |
   | 社区生态  | 15%  | 5（说明） | 3（说明） | 4（说明） |
   | 迁移风险  | 10%  | 2（说明） | 5（说明） | 3（说明） |
   | **加权总分** | — | **3.65** | **3.75** | **3.35** |

   ### 方案深度分析

   #### 方案 A
   - **核心优势**：……（列举得分最高维度及原因）
   - **致命弱点**：……（列举得分最低维度及影响）
   - **最适用于**：……（描述方案 A 发挥优势的具体场景）

   #### 方案 B
   （同上格式）

   ### 初步推荐
   > 基于当前权重设置，**方案 B** 加权总分最高（3.75）。
   > 若您最看重性能，则 **方案 A** 更优。
   > 建议结合 `decision-framework` 技能做最终决策。
   ```

## 示例

### 示例一：消息队列技术选型

**触发提示词**：

```
我们要为订单系统引入消息队列，候选方案是 Kafka、RabbitMQ 和 RocketMQ。
团队 Java 技术栈，最看重可靠性和运维难度，预算有限，请帮我对比
```

**执行前状态**：用户提供了 3 个候选方案和优先级偏好（可靠性 > 运维难度 > 成本）。

**预期输出**：

```markdown
## 方案对比报告

### 候选方案概述
- **Kafka**：Apache 开源分布式流处理平台，高吞吐量，适合日志/事件流场景
- **RabbitMQ**：AMQP 协议消息代理，功能丰富，支持复杂路由，社区成熟
- **RocketMQ**：阿里巴巴开源，专为金融级可靠性设计，对 Java 生态友好

### 评估矩阵

| 评估维度 | 权重 | Kafka | RabbitMQ | RocketMQ |
|---------|------|-------|---------|---------|
| 消息可靠性 | 35% | 4（副本机制强，但需调优） | 3（支持持久化，但吞吐下降） | 5（金融级可靠性，事务消息原生支持） |
| 运维难度 | 30% | 2（需维护 ZooKeeper/KRaft，学习曲线陡） | 4（管理界面友好，文档丰富） | 3（阿里云托管版可简化，自建较复杂） |
| Java 生态兼容 | 20% | 4（Spring Integration 支持好） | 4（Spring AMQP 成熟） | 5（原生 Java SDK，与 Spring 深度集成） |
| 许可证/成本 | 15% | 3（开源，但云厂商支持需付费） | 4（完全开源免费） | 4（开源，阿里云有商业版） |
| **加权总分** | — | **3.25** | **3.65** | **4.25** |

### 方案深度分析

#### RocketMQ（加权总分：4.25）
- **核心优势**：消息可靠性最高（支持事务消息、顺序消息、延迟消息），Java 原生友好
- **致命弱点**：国内社区为主，英文文档相对薄弱；自建运维有一定复杂度
- **最适用于**：订单、支付等对消息丢失零容忍的金融交易场景

### 初步推荐
> 基于当前权重，**RocketMQ** 加权总分最高（4.25），且与订单系统的可靠性需求高度匹配。
> 建议结合 `decision-framework` 技能确认最终选择。
```

### 示例二：缓存策略对比

**触发提示词**：

```
商品详情页需要加缓存，对比：本地内存缓存（Caffeine）、Redis、CDN 静态化，我们最在意响应速度和数据一致性
```

**预期输出**：建立包含响应速度、数据一致性、运维成本、适用数据类型四个维度的对比矩阵，Caffeine 在响应速度得 5 分但数据一致性得 2 分，Redis 均衡，CDN 静态化在响应速度和成本上表现优异但数据一致性仅 1 分，最终报告说明三种方案可组合使用（CDN + Redis 双层缓存）。

## 注意事项

- 评估维度和权重强烈依赖业务场景，用户应在确认权重后再执行打分，避免方向性偏差
- 打分基于通用领域知识和用户提供的信息，对于特定公司内部系统（如私有云能力），用户应根据实际情况调整分数
- 加权总分仅是参考，应结合"致命弱点"做排除性判断——即使某方案总分第一，若其致命弱点恰好是不可接受的风险，应直接排除
- 可与 `logical-reasoning` 技能配合：先用推理技能确认评估维度是否遗漏，再用本技能打分
- 可与 `decision-framework` 技能配合：本技能输出的对比矩阵可直接作为决策框架的输入
