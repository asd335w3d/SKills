## 说明

本技能提供结构化的决策判断框架，将目标澄清、约束过滤、方案评估、风险识别和推荐理由五个阶段串联为一套可复用的决策流程。无论是技术架构选型、产品功能取舍还是工程路径规划，均可通过本技能将"我应该选哪个"这一模糊问题，转化为"在 X 目标和 Y 约束下，A 方案在 Z 维度上优于 B 方案，主要风险可通过 M 措施缓解，因此推荐 A"这样的结构化决策结论。

**适用场景**：

- 场景一：技术方案已用 `solution-comparison` 比较完毕，需要综合比较结果、团队能力和业务时间窗口，做出并对外宣讲最终选择
- 场景二：需求讨论出现多方意见分歧，需要一个中立的结构化框架帮助团队聚焦在目标和约束上，而非争论个人偏好
- 场景三：面对"维持现状 vs 小改 vs 大改"三选一的典型决策，需要综合短期成本和长期收益做出取舍
- 场景四：需要向管理层或跨团队汇报技术决策，输出一份结构清晰、有理有据的决策文档

**前提条件**：

- 用户需提供决策的背景、目标和候选方案（可以是自然语言描述，也可以是 `solution-comparison` 的比较报告）
- 若有硬性约束（预算、时间、团队规模、监管合规要求等），必须明确提供

**不适用场景**：

- 不适用于纯情感或价值观层面的决策（本框架基于逻辑和事实）
- 决策建议基于提供的信息，若信息严重不完整，结论将标注"信息不足，建议先收集数据"而非强行给出推荐

## 流程

1. **澄清决策目标**：从用户描述中提取决策的核心目标，并按优先级排列：
   - **首要目标**（必须实现，不可妥协）
   - **次要目标**（应尽量实现，可在方案间权衡）
   - **期望目标**（可选，有则更好）
   
   若用户未明确区分优先级，主动提问：「请确认以下目标的优先级：……」

2. **识别约束条件**：区分以下两类约束：
   - **硬性约束**（Hard Constraints）：必须满足，不满足则直接淘汰方案。例：「上线时间不超过 3 个月」「必须符合 GDPR」「不引入新的外部依赖」
   - **软性约束**（Soft Constraints）：尽量满足，可在方案间权衡。例：「优先使用团队熟悉的技术」「尽量减少运维负担」
   
   对每个约束标注来源（用户明确提及 / 从背景推断）。

3. **过滤不满足硬性约束的方案**：逐一检查每个候选方案是否满足所有硬性约束，不满足者直接淘汰，并说明淘汰原因。若所有方案均被淘汰，停止并提示用户放宽约束或增加候选方案。

4. **综合评估剩余方案**：对通过约束过滤的方案，综合以下维度进行评判：
   - **目标契合度**：方案对首要目标、次要目标的实现程度（高/中/低）
   - **软性约束满足度**：满足软性约束的数量和程度
   - **方案对比得分**（若已执行 `solution-comparison`，直接引用其加权总分）
   - **实施可行性**：团队现有能力、资源是否支持方案落地

5. **识别并量化风险**：对每个候选方案，识别主要风险并评估：

   ```
   风险 = 发生概率（高/中/低） × 影响程度（高/中/低）
   风险等级：
     - 高概率 × 高影响 = 🔴 不可接受风险
     - 中概率 × 高影响 / 高概率 × 中影响 = 🟡 需要缓解措施
     - 低概率 × 任何影响 / 任何概率 × 低影响 = 🔵 可接受风险
   ```

   对每个 🔴 和 🟡 风险，提供具体的**缓解措施**。

6. **做出最终推荐**：综合以上分析，按以下逻辑给出推荐：
   - 若明确存在一个方案在首要目标上领先且无 🔴 风险 → 直接推荐
   - 若两个方案势均力敌 → 给出条件推荐（「若更看重 X 则选 A，若更看重 Y 则选 B」）
   - 若所有方案均有 🔴 风险 → 建议先制定缓解措施，再推荐

7. **生成决策报告**：按以下格式输出，可直接用于团队会议或向上汇报：

   ```markdown
   ## 决策报告：<决策标题>

   ### 决策背景
   （1～3 句，说明做此决策的背景和时间压力）

   ### 目标与约束
   **首要目标**：……
   **次要目标**：……
   **硬性约束**：……
   **软性约束**：……

   ### 候选方案筛选
   | 方案 | 是否通过硬性约束 | 淘汰原因（如有） |
   |-----|--------------|--------------|
   | 方案 A | ✅ 通过 | — |
   | 方案 B | ❌ 淘汰 | 不满足约束：…… |

   ### 方案综合评估

   | 评估维度 | 方案 A | 方案 C |
   |---------|--------|--------|
   | 首要目标契合度 | 高 | 中 |
   | 次要目标契合度 | 中 | 高 |
   | 方案对比加权分 | 4.25 | 3.65 |
   | 实施可行性 | 中（需学习曲线） | 高 |
   | 主要风险等级 | 🟡（可缓解） | 🔵（可接受） |

   ### 风险评估

   #### 方案 A 的主要风险
   - 🟡 **团队学习曲线**（中概率 × 高影响）
     缓解措施：安排 2 周培训 + 指定内部技术负责人
   - 🔵 **第三方依赖更新**（低概率 × 中影响）
     缓解措施：锁定版本，定期评估升级计划

   ### 最终推荐

   > **推荐选择：方案 A**
   >
   > **核心理由**：
   > 1. 方案 A 在首要目标（……）上显著优于其他方案
   > 2. 唯一主要风险（学习曲线）可通过培训计划有效缓解，不构成决策障碍
   > 3. 与现有技术栈兼容，长期维护成本可控
   >
   > **替代方案**：若项目时间窗口缩短至 1 个月以内，应切换至方案 C（实施可行性更高，但在 X 维度上接受一定妥协）
   >
   > **决策有效期**：建议在 6 个月后根据业务发展重新评估

   ### 实施路径建议
   1. **第 1 步**（第 1～2 周）：……
   2. **第 2 步**（第 3～4 周）：……
   3. **验收标准**：……
   ```

## 示例

### 示例一：数据库迁移决策

**触发提示词**：

```
我们的 MySQL 单库撑不住了，考虑三个方向：
1. MySQL 分库分表（ShardingSphere）
2. 迁移到 TiDB
3. 业务拆分 + 多个独立 MySQL 实例
团队 5 人，均熟悉 MySQL，不熟悉 TiDB，上线时间最多 4 个月，请帮我做最终决策
```

**执行前状态**：用户提供了 3 个候选方案、团队背景和时间约束。

**预期输出**：

```markdown
## 决策报告：数据库扩容方案选择

### 目标与约束
**首要目标**：解决数据库性能瓶颈，支撑业务增长
**硬性约束**：上线时间 ≤ 4 个月；团队不熟悉 TiDB（学习成本纳入评估）
**软性约束**：尽量沿用现有 MySQL 运维经验；尽量减少应用层改造量

### 候选方案筛选
| 方案 | 是否通过硬性约束 | 说明 |
|-----|--------------|------|
| MySQL 分库分表 | ✅ 通过 | — |
| TiDB | 🟡 风险 | 团队零经验，4 个月内完成迁移风险较高 |
| 业务拆分 + 多 MySQL | ✅ 通过 | — |

> TiDB 未被直接淘汰，但标记为高风险方案，需在后续风险评估中重点评估。

### 最终推荐

> **推荐选择：业务拆分 + 多个独立 MySQL 实例**
>
> **核心理由**：
> 1. 完全沿用团队熟悉的 MySQL 运维技能，实施风险最低
> 2. 业务拆分同时改善服务边界，为后续微服务演进打基础
> 3. 在 4 个月时间约束内，该方案最可能按时交付
>
> **替代方案**：若业务拆分改造量过大（超过 2000 行应用代码变更），切换至分库分表方案
>
> **实施路径建议**：
> 1. 第 1～2 周：梳理业务模块边界，识别数据耦合点
> 2. 第 3～6 周：优先拆分读写负载最高的模块（通常为订单/用户）
> 3. 验收标准：P99 查询延迟低于 100ms，单库容量低于 500GB
```

### 示例二：有既有比较报告时的决策

**触发提示词**：

```
我已经用 solution-comparison 对比了 Kafka、RabbitMQ 和 RocketMQ，
RocketMQ 总分最高（4.25），但我们 CTO 担心它的社区规模，请帮我做最终决策
```

**预期输出**：引入 CTO 担忧作为新的软性约束，重新评估三个方案，若 RocketMQ 的社区风险可通过"使用阿里云托管版降低自建依赖"来缓解，则维持 RocketMQ 的推荐；若团队无法接受这一风险，则推荐 RabbitMQ 作为替代，并说明在可靠性维度上需要额外的工程投入来弥补差距。

## 注意事项

- 决策框架的输出质量高度依赖目标和约束的清晰度，模糊的目标（如"尽量好"）会导致决策结果缺乏说服力，执行前应要求用户明确化
- 本框架提供决策建议，最终决策权在用户和团队；当技术建议与组织政治/文化因素冲突时，应在报告中明确标注"此因素超出本框架分析范围"
- "决策有效期"字段非常重要：技术决策应定期复查，避免在环境变化后仍执行过时的决策
- 建议完整使用推荐链路：先用 `logical-reasoning` 厘清问题，再用 `solution-comparison` 量化对比，最后用本技能做最终判断和输出决策文档
